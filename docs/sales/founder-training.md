# Founder-тренажёр — научиться объяснять и продавать Platform API

Цель: не стать налоговым юристом, а **уметь объяснить compliance-flow бизнеса**
и пройти весь сценарий руками. Когда пройдёшь сама — сможешь продавать.

---

## Метод — как учиться (5 шагов)

1. **Прочитай Часть 1** — мысленная модель (20-30 минут). Без неё практика
   будет «нажимаю кнопки, не понимаю».
2. **Пройди Часть 2 руками** — весь сценарий в кабинете, пока не пройдёшь
   без подсказок. Не читать дальше, пока не сделала.
3. **Проговори Часть 3 вслух** — demo на 10-15 минут. Запиши себя на телефон,
   послушай, повтори 3-5 раз, пока не идёт без бумажки.
4. **Веди FAQ (Часть 4)** — после каждого вопроса клиента записывай ответ сюда.
5. **Повторяй цикл.** Ты строишь, продаёшь и интегрируешь одновременно — это
   самый быстрый способ стать экспертом.

# ЧАСТЬ 1 — Мысленная модель

## 1.1 Business flow — жизнь одной выплаты

```
ЗАКАЗ  ->  ПРОВЕРКА  ->  РЕШЕНИЕ  ->  ВЫПЛАТА  ->  ФИСКАЛИЗАЦИЯ  ->  СТАТУС
```

1. **Заказ** — курьер выполнил доставку, платформа должна ему заплатить.
2. **Проверка** — кто этот курьер? Валиден ли ИИН? Какой у него статус
   (самозанятый / ИП / ТОО)? Не превысит ли он лимит 300 МРП в этом месяце?
3. **Решение** — `PROCEED` (платить можно) / `BLOCK` (нельзя) /
   `WARNING` (можно, но есть нюанс).
4. **Выплата** — платформа переводит деньги курьеру **сама, своим банком**,
   и удерживает 4% соцплатежей.
5. **Фискализация** — чек оформляется и регистрируется в КГД.
6. **Статус** — платформа видит фискальный номер, чек закрыт.
7. *(Отмена)* — если заказ отменили до фискализации — откат, лимит
   возвращается.

Запомни ключевое: **деньги через нас не идут.** Мы — «мозг», который говорит
можно/нельзя и оформляет чек. Деньги двигает сама платформа.

## 1.2 Главные риски клиента — и что их закрывает

| Риск платформы | Чем грозит | Что закрывает |
|---|---|---|
| Выплата без фискализации | штраф от КГД | фискализация (встроена в выплату) |
| Превышение лимита 300 МРП | курьер незаконно остаётся самозанятым — **риск на платформе** | контроль лимита |
| Выплата на неверный/чужой ИИН | деньги не туда, проблемы с учётом | валидация ИИН |
| Выплата на ТОО как на курьера | неправомерная операция | проверка статуса налогоплательщика |
| Нет чеков / нет доказательств | нечем отчитаться перед проверкой КГД | реестр чеков + аудит-журнал |
| Неверные удержания | недоплата/переплата налогов | единый flow `/process-payment` |

## 1.3 Что делает Esep — в одном абзаце

> Платформа на каждую выплату спрашивает наш API: «этому человеку можно
> столько заплатить — и оформи это по закону». Мы проверяем ИИН, статус и
> лимит, оформляем фискальный чек, ведём учёт. Платформа просто читает ответ.
> Вся работа с КГД и оператором фискальных данных спрятана за одним
> HTTP-запросом.

Если можешь сказать это своими словами без бумажки — Часть 1 усвоена.

# ЧАСТЬ 2 — Практика: пройди сценарий руками

Открой **https://api.esepkz.com/platform.html** и войди в демо-кабинет
курьерской службы (логин/пароль демо-доступа — у тебя есть; если нет —
создаётся скриптом `server/scripts/seed_demo_courier_client.js`).

Тестовый ИИН курьера: **850101100012** (это валидный тестовый ИИН — проходит
алгоритм проверки; реальному человеку не принадлежит).

Идём по шагам. На каждом: **что сделать → что увидишь → что это значит.**

### Шаг A. Проверь курьера

- **Сделать:** карточка «Валидация ИИН» → ввести `850101100012` → Выполнить.
- **Увидишь:** `valid: true`, дата рождения, пол.
- **Значит:** ИИН настоящий по контрольной цифре. Так платформа ловит
  опечатки до выплаты. Попробуй ввести `999999999999` — увидишь красный отказ.

### Шаг B. Посмотри лимит ДО выплаты

- **Сделать:** карточка «Лимит 300 МРП» → ввести `850101100012` → Выполнить.
- **Увидишь:** «использовано: X ₸», «осталось: …».
- **Значит:** это стартовая точка. **Запиши число «использовано».**

### Шаг C. Проведи выплату

- **Сделать:** карточка «Выплата курьеру» → ИИН `850101100012`, сумма `50000`,
  order_id оставь сгенерированный. → Выполнить.
- **Увидишь:** `decision`, проверки, `fiscal_status`.
- **Запиши order_id** — он нужен дальше.
- **Значит:** один вызов сделал всё — проверку, учёт, постановку чека.
  *Примечание: на демо-проде фискализация в режиме очереди, поэтому
  `decision` может быть `WARNING` с пометкой «фискализация отключена». На
  боевом клиенте с подключённой кассой будет чистый `PROCEED`.*

### Шаг D. Посмотри лимит ПОСЛЕ выплаты

- **Сделать:** снова «Лимит 300 МРП» → `850101100012`.
- **Увидишь:** «использовано» выросло на 50 000 ₸.
- **Значит:** выплата учтена. Так работает контроль лимита — нарастающим итогом.

### Шаг E. Проверь статус чека

- **Сделать:** карточка «Статус фискального чека» → вставь order_id из шага C.
- **Увидишь:** статус чека (в очереди / фискализирован / отменён).
- **Значит:** так платформа узнаёт, что с чеком. После доставки и пробития
  чека здесь появится фискальный номер и QR.

### Шаг F. Отмени заказ и проверь откат

- **Сделать:** карточка «Аннулирование заказа» → вставь тот же order_id →
  Выполнить. Затем снова открой «Лимит 300 МРП» по `850101100012`.
- **Увидишь:** заказ отменён; «использовано» вернулось к числу из шага B.
- **Значит:** отмена откатывает учёт лимита. Курьер не «потерял» лимит
  из-за отменённого заказа.

**Если прошла все 6 шагов и понимаешь, что произошло на каждом — ты готова
показывать продукт.**

# ЧАСТЬ 3 — Founder demo (10-15 минут)

Сценарий для звонка с клиентом. Показываешь экран (кабинет), говоришь.

**Минута 0-2 — проблема.** «С 2026 по НК РК платформа, которая платит
курьерам, — налоговый агент. Вы обязаны фискализировать каждую выплату,
удерживать соцплатежи и следить, чтобы курьер не превысил 300 МРП в месяц.
Без этого — штрафы.»

**Минута 2-4 — идея.** «Мы закрываем это одним API. Все договоры с КГД и
оператором фискальных данных — на нашей стороне. Вы делаете один
HTTP-запрос на выплату.»

**Минута 4-10 — показываешь flow вживую** (Часть 2):
- «Курьер выполнил заказ. Платформа вызывает наш API — вот этот вызов.»
  *(карточка «Выплата курьеру»)*
- «Мы за миллисекунды: проверили ИИН, статус, лимит 300 МРП, оформили чек.»
- «Платформа получает одно решение — можно платить или нет.» *(decision)*
- «Лимит ведётся автоматически.» *(карточка лимита — до/после)*
- «Статус чека платформа читает вот так.» *(карточка статуса)*
- «Заказ отменили — лимит откатывается.» *(аннулирование)*

**Минута 10-13 — почему мы.** «Без нас: договор с КГД, договор с ОФД, своя
разработка, штат compliance. С нами: интеграция 1-2 дня, один заголовок.»

**Минута 13-15 — следующий шаг.** «Предлагаю пилот на 2-4 недели на
ограниченной группе курьеров. Какие у вас вопросы?»

Главное правило demo: **показываешь flow, а не код.**

# ЧАСТЬ 4 — FAQ (заготовка — дополняй после каждого звонка)

| Вопрос | Ответ |
|---|---|
| Это платёжный процессинг? | Нет. Деньги вы платите сами. Мы проверяем, оформляем чек, ведём учёт. |
| Нужен ли нам договор с КГД / ОФД? | Нет. Все договоры на нашей стороне. Вы интегрируетесь только с нами. |
| Сколько занимает интеграция? | 1-2 рабочих дня одному разработчику. |
| Кто платит налог за курьера? | Платформа удерживает соцплатежи и перечисляет. У самозанятого ИПН = 0%, 4% — это соцплатежи (ОПВ/ОПВР/СО/ОСМС по 1%). |
| Что если у нас P2P-модель (клиент платит курьеру напрямую)? | Тогда удержать налог нельзя — продукт не подойдёт. Подходит транзакционная модель. |
| Что если курьер работает на двух платформах? | Мы видим доход только через наш API. Полную картину даст реестр КГД (подключаем). |
| Что если Webkassa недоступна? | Чек помечается как «ошибка загрузки», операцию можно повторить — учёт не теряется. |
| Можно ли отменить уже фискализированный чек? | Через возвратный чек — отдельная процедура. До фискализации — обычная отмена с откатом лимита. |
| Повторный вызов с тем же order_id задвоит выплату? | Нет. API идемпотентен — повторный вызов вернёт прежний результат. |
