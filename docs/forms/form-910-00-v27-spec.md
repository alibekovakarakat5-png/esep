Источник: официальный пакет СОНО с kgd.gov.kz
(`form_910_00_v27_r133.tar.bz2`, обновлён 30.12.2025).
Извлечено из определения формы для переписывания `lib/core/services/form910_service.dart`.

> ⚠️ Пакет даёт **имена полей и метаданные формы**, но НЕ даёт точную
> обёртку XML-экспорта (корневой конверт). Для конверта нужен реальный
> образец заполненного экспорта из СОНО — его пришлёт пользователь.

## Основная форма 910.00

Поля `field_910_00_001` … `field_910_00_026`.
Многие имеют месячную разбивку — суффиксы `_1` … `_6` (6 месяцев полугодия)
и буквенные подполя (`_A`, `_B`, `_A_I`, `_A_II`, `_016_A..E`, `_026_a..c`).

## Два формата: XML (СОНО) vs JSON (ИСНА)

Проверено по kgd.gov.kz, pro1c.kz, uchet.kz, 1cbit.kz:
- **СОНО** (XML) — старая система, **сворачивается**. Пакет формы выше — из неё.
- **КНП ИСНА** (JSON) — новая система, **актуальная на 2026**. Форма 910 уже в ИСНА.
  1С выгружает JSON с версии 3.0.69.1 («Экспорт в ИСНА (JSON)»).
- ⚠️ JSON-схема ИСНА **публично не опубликована** — ни КГД, ни форумы её не дают.

`form910_service.dart` реализует **оба** формата (`Form910Format.xmlSono` /
`jsonIsna`) с **реальными именами полей**, но:
- конверт XML — по догадке (структура `<form code=...>` + `field_*`)
- структура JSON — по догадке (`{formCode, period, taxpayer, fields:{...}}`)

## Что ещё нужно для боевого экспорта

1. **Реальный образец экспорта** — JSON из 1С/КНП ИСНА (приоритет) и/или
   XML из СОНО. Нужен, чтобы заменить угаданные конверты на настоящие.
2. **Помесячная модель данных** — форма ждёт разбивку по 6 месяцам
   (`_1.._6`), а `Form910Service` считает 6-месячные агрегаты. Нужно
   тянуть помесячные данные из транзакций.
3. После п.1-2 — поправить конверты в `generateXml()` / `generateJson()`
   (имена полей уже официальные, менять не придётся).
