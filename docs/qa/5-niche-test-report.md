## Метод

1. В БД созданы 5 enterprise-клиентов под разные ниши — каждый со своим API-ключом, БИН и набором фич.

## 🔴 КРИТИЧЕСКИЙ БАГ — `/cancel-order` не откатывает учёт лимита

**Симптом.** После cancel у всех 5 ниш `getMonthlyIncome` для исполнителя продолжает считать отменённую выплату. Прирост `used` между нишами +100 000 ₸ (60k подтверждённая + 40k «отменённая»), а должен быть +60 000 ₸ (только подтверждённая).

**Причина в коде.** `server/src/routes/platform/cancel_order.js`, строки 100-109:

```js
await db.query(
  `UPDATE platform_self_employed_income
      SET note = COALESCE(note, '') || ' [CANCELLED ' || NOW()::TEXT || ']'
    WHERE external_id = $1
      AND api_key_id = $2`,
  [order_id, req.platformClient.id],
);
```

Код только дописывает `[CANCELLED ...]` в `note`, **не меняя `amount` и не помечая запись «исключённой из лимита»**. А `getMonthlyIncome` делает `SELECT SUM(amount)` без фильтра — поэтому отменённые выплаты продолжают съедать лимит 300 МРП.

**Что обещано клиенту (но не работает):**
- `docs/sales/founder-training.md` Часть 2 шаг F: «использовано вернулось к числу из шага B».
- `docs/PLATFORM-API.md` сервис #5: «soft cancel, **откат учёта лимита**».
- Капабилити-PDF: «помечает заказ отменённым и **откатывает учёт лимита**».

**Влияние.** Курьер, заказы которого отменили несколько раз, получит ложный `BLOCK` на следующих выплатах, хотя по факту он в лимите. На пилоте у клиента это вскроется в первый же день.

**Фикс (правильный путь — TDD):**
1. Добавить колонку `cancelled_at TIMESTAMPTZ` в `platform_self_employed_income`.
2. На cancel — `UPDATE ... SET cancelled_at = NOW()` (вместо append к note).
3. В `getMonthlyIncome` — фильтр `WHERE cancelled_at IS NULL`.
4. Тест 13.6 в e2e: после cancel `used` уменьшается на сумму отменённой выплаты.

Объём фикса: 15-30 минут. Срочность: до пилота — **обязательно**.

## 🟡 Средние находки

### Feature flag `fiscalize` — рудимент

`/process-payment` фискализирует независимо от того, есть ли у клиента фича `fiscalize` в `features[]` (она была для legacy `/fiscalize/issue`, который теперь stub). У ниши «Услуги Beauty Pro» в тесте `fiscalize` НЕ был включён — но `process-payment` всё равно вернул `fiscal_status: awaiting_courier_fiscalization` (Webkassa-загрузка прошла).

**Что делать:** либо убрать флаг `fiscalize` (он ни на что не влияет, путает биллинг), либо реально гейтить фискализацию в `/process-payment` по этому флагу (если хотим тарифицировать «фискализация — отдельно»). Решение зависит от бизнес-модели.

### Кросс-тенант ИИН — feature, не bug

Лимит 300 МРП считается по ИИН **глобально** через нашу БД, не на одного клиента. В тесте я использовала один тест-ИИН для 5 разных «компаний» — лимит съедался кумулятивно. Это **корректно** по закону (НК РК 2026 — лимит на самозанятого, не на платформу), и это наш **selling point**: если курьер работает на двух наших клиентах-платформах, мы видим суммарно и защищаем обоих. Но на discovery-звонке стоит проговорить, чтобы клиент не удивился.

## Как клиент интегрируется (для разработчика клиента)

1. **Получает API-ключ** от менеджера Esep.
2. **Добавляет заголовок** `X-Platform-Key: <ключ>` ко всем запросам к `https://api.esepkz.com/api/platform/*`.
3. **Smoke-test** — `GET /me` должен вернуть имя клиента и список фич.
4. **Встраивает в момент выплаты:**
   ```
   POST /api/platform/process-payment
   { "courier_iin": "<ИИН>", "amount": <тенге>, "order_id": "<уникальный ID>" }
   ```
   По полю `decision`:
   - `PROCEED` → выплачивать
   - `WARNING` → выплачивать, но показать менеджеру предупреждение
   - `BLOCK` → НЕ выплачивать (вывести причину из `errors[]`)
5. **Хранит у себя** order_id и ИИН — для последующих запросов статуса/отмены.
6. **Опционально**: периодически дёргает `GET /receipts/:order_id` для отслеживания фискализации (или ждёт нашего outbound-webhook, когда сделаем).
7. **При отмене заказа** — `POST /cancel-order` с тем же order_id.

**Оценка трудозатрат разработчика клиента:** 1-2 рабочих дня на базовую интеграцию (process-payment + receipt-status). Полная интеграция со всеми сервисами и обработкой ошибок: 3-5 дней.
